<objective>
Create durable operations documentation for runtime signal routing, low-noise alerts, the operator dashboard/checklist, and least-privilege operational access.
</objective>

## Context

Plan 11-01 establishes code-level webhook operational events and the phase evidence ledger. This plan turns the source-level signals and Phase 9 blocked evidence into operator-facing documentation without pretending live provider setup already exists.

## Recommendation

Keep the operations doc practical and provider-neutral. Use required-before-Phase-12 placeholders for provider links, notification paths, and live alert proof.

## Artifacts This Plan Produces

- `docs/OPERATIONS.md`
- Updated `.planning/phases/11-operational-monitoring-alerting-and-incident-readiness/11-OPERATIONS-EVIDENCE.md`
- Optional `docs/DEPLOYMENT.md` link to `docs/OPERATIONS.md` if no operations cross-link exists

<threat_model>

### T-11-04: Alert Noise or Unowned Monitoring

Risk: broad alerts are documented but no role owns the response, causing either alert fatigue or missed incidents.

Mitigation: every alert row must include owner role, severity, notification path/placeholder, first response action, and false-positive/noise guidance.

### T-11-05: Provider Ambiguity Becomes Fake Evidence

Risk: documentation includes invented dashboard URLs or provider-specific paths.

Mitigation: use provider-neutral placeholders marked `required-before-Phase-12` unless the value comes from Phase 9 safe evidence.

### T-11-06: Access Matrix Leaks Account or Secret Details

Risk: access documentation exposes account IDs, dashboard IDs, secret names with values, API keys, connection strings, or private URLs.

Mitigation: record role, surface, least-privilege expectation, evidence status, and blocker only; never commit secret-bearing account details.

</threat_model>
