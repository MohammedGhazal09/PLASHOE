<objective>
Make Stripe webhook outcomes visible as safe structured operational events and start the Phase 11 evidence ledger without fabricating live-provider proof.
</objective>

## Context

Phase 8 already provides structured JSON logging, redaction, request ids, health, readiness, startup logs, and deployment guidance. Phase 11 only needs a focused webhook logging fix before the broader operations docs can reference the webhook signal confidently.

## Recommendation

Do this first because the alert catalog and incident runbook should be based on safe event names that exist in code and tests.

## Artifacts This Plan Produces

- `emitStripeWebhookLog` or equivalent local helper in `Backend/controllers/webhookController.js`.
- Structured event names: `stripe-webhook-invalid-signature`, `stripe-webhook-duplicate`, `stripe-webhook-accepted`, `stripe-webhook-processing-failed`.
- Focused webhook logging assertions in `Backend/test/payment-webhook.test.js`.
- `.planning/phases/11-operational-monitoring-alerting-and-incident-readiness/11-OPERATIONS-EVIDENCE.md`.

<threat_model>

### T-11-01: Secret or Payload Leakage Through Webhook Logs

Risk: webhook logs accidentally include Stripe signatures, webhook secrets, raw payloads, request bodies, bearer tokens, MongoDB URIs, passwords, stacks, or raw PII.

Mitigation: reuse `serializeError` and the existing structured logger redaction; log only allowlisted metadata fields; add tests that inspect emitted JSON log records for safe event names and absence of forbidden values.

### T-11-02: Operational Signal Gap Remains

Risk: alerts and runbooks reference webhook events that do not exist in code.

Mitigation: add event-name assertions for invalid signature, duplicate, accepted, and processing-failed paths before writing provider-facing alert docs.

### T-11-03: False Evidence

Risk: Phase 11 evidence looks complete even though Phase 9 staging/provider inputs are still missing.

Mitigation: initialize the evidence ledger with explicit `blocked` rows naming missing Phase 9 inputs.

</threat_model>

<action>

Update webhook logging in `Backend/controllers/webhookController.js`:

- Import `logInfo`, `logWarn`, `logError`, and `serializeError` from `../utils/logger.js`.
- Add a small local helper such as `getWebhookLogMetadata({ req, event, eventRecord, duplicate, status, error, signaturePresent })`.
- The helper may return only these fields when present: `requestId`, `eventId`, `eventType`, `status`, `duplicate`, `signaturePresent`, and `error: serializeError(error)`.
- In the invalid-signature catch block, emit `logWarn("stripe-webhook-invalid-signature", { requestId: req.requestId, signaturePresent: Boolean(signature) })` before returning `400`.
- In the duplicate branch, emit `logInfo("stripe-webhook-duplicate", { requestId: req.requestId, eventId: event.id, eventType: event.type, duplicate: true, status: claim.eventRecord?.status || "unknown" })` before returning success.
- After `markPaymentEventProcessed`, emit `logInfo("stripe-webhook-accepted", { requestId: req.requestId, eventId: event.id, eventType: event.type, status: "processed", duplicate: false })`.
- In the processing-failure catch block, emit `logError("stripe-webhook-processing-failed", { requestId: req.requestId, eventId: event?.id, eventType: event?.type, status: "failed", error: serializeError(error) })`.
- Remove `console.error(error?.stack || error)`.
- Do not log `req.body`, the Stripe signature value, `STRIPE_WEBHOOK_SECRET`, raw Stripe object payloads, full request bodies, stack traces, bearer tokens, MongoDB URIs, passwords, or customer/user fields.

</action>

  - accepted event -> `stripe-webhook-accepted`

- Keep existing webhook behavior assertions unchanged: invalid signatures return `400`, duplicates are success no-ops, unresolved events return `500`, and failed events persist status `failed`.
