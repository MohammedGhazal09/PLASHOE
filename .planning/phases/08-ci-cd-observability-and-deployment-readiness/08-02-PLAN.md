<objective>
Add deployment-grade backend liveness/readiness behavior and safe structured operational logs with request correlation.
</objective>

## Context

- `Backend/app.js` exports the Express app and currently exposes `/api/health`.
- `Backend/server.js` validates runtime env, connects MongoDB, and starts the listener.
- `Backend/config/db.js` logs prose connection messages and continues after connection failure.
- `Backend/middleware/security.js` owns final security/application error envelopes.
- Backend tests already use MongoMemoryReplSet, Supertest, and Vitest.

## Recommendation

Implement observability as small local helpers and middleware. Keep signals explicit and testable, avoid new runtime dependencies, and never log sensitive request/body/secret data.

## Artifacts This Plan Produces

- `Backend/utils/readiness.js`
- `Backend/utils/logger.js`
- `Backend/middleware/requestContext.js`
- Updated `Backend/app.js`, `Backend/server.js`, `Backend/config/db.js`, and `Backend/middleware/security.js`
- Updated backend tests and API docs

<threat_model>

### T-08-04: False Healthy Deployments

Risk: deployment platforms treat `/api/health` as proof while MongoDB-backed routes are unusable.

Mitigation: keep `/api/health` as liveness and add `/api/ready` with Mongoose readiness state.

### T-08-05: Sensitive Log Leakage

Risk: operational logs accidentally expose JWTs, Stripe secrets, MongoDB URIs, passwords, raw webhook payloads, or full request bodies.

Mitigation: centralize redaction and test representative sensitive values.

### T-08-06: Untraceable Production Errors

Risk: support cannot connect a user-visible error to logs.

Mitigation: add request ids to response headers, error envelopes, completion logs, and error logs.

</threat_model>
