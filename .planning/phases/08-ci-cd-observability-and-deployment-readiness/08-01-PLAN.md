<objective>
Create the Phase 08 CI gate that installs both nested apps, runs the existing local verification commands, and applies the documented production audit policy.
</objective>

## Context

- The repository has no `.github/workflows` directory.
- The locked Phase 08 spec asks for GitHub Actions on pull requests and pushes to `main`.
- Backend tests run from `Backend` with `npm test`.
- Frontend tests/build run from `Frontend/Ecommerce-main/my-app`.
- The static contract checker runs from the repository root.
- Frontend production audit debt from Create React App tooling is already accepted in the Phase 03 security risk register.
- Current external research shows Node 20 is EOL in June 2026, but the locked Phase 08 spec still says Node 20. Preserve the locked runtime unless the spec is amended; record the drift in verification.

## Recommendation

Implement one readable CI workflow with separate jobs and a small audit script. Keep the workflow scoped to Phase 08 and do not add deployment, CodeQL, ZAP, Lighthouse, Docker, or coverage thresholds.

## Artifacts This Plan Produces

- `.github/workflows/ci.yml`
- `scripts/ci/check-audits.mjs`
- Updated CI/audit sections in `docs/TESTING.md`

<threat_model>

### T-08-01: CI Does Not Match Local Verification

Risk: CI gives false confidence if it skips the exact backend/frontend/static commands used through Phase 07.

Mitigation: encode the existing commands directly and verify them statically and locally.

### T-08-02: Accepted Frontend Risk Becomes Hidden

Risk: suppressing frontend audit failures makes new production dependency risk invisible.

Mitigation: run frontend audit visibly and compare findings to the accepted CRA/tooling boundary.

### T-08-03: Overbroad Workflow Permissions

Risk: default or write-capable workflow permissions create unnecessary supply-chain risk.

Mitigation: set top-level `permissions: contents: read` and do not request deploy/package/id-token scopes.

</threat_model>

<action>

Create `.github/workflows/ci.yml` with:

- `name: CI`
- `on.pull_request.branches: [main]`
- `on.push.branches: [main]`
- workflow-level `permissions: contents: read`
- workflow-level `concurrency` keyed by workflow and PR head ref/ref, with `cancel-in-progress: true`
- `backend` job on `ubuntu-latest`, using official checkout/setup-node major versions, Node version locked by Phase 08, npm cache, `cache-dependency-path: Backend/package-lock.json`, `npm ci`, and `npm test`
- `frontend` job on `ubuntu-latest`, using npm cache for `Frontend/Ecommerce-main/my-app/package-lock.json`, `npm ci`, `npm test -- --watchAll=false`, and `npm run build`
- `static-contracts` job that checks out the repo and runs `node .planning/spikes/001-core-flow-contract-check/check-contracts.mjs`
- `audit-policy` job that runs `node scripts/ci/check-audits.mjs`

Use `working-directory` instead of long shell chains where it keeps YAML readable. Do not add deployment jobs.

</action>

<task id="08-01-02" type="code">
<files>

- `scripts/ci/check-audits.mjs`

</files>
<action>

Add a Node script that runs production dependency audits for both nested apps and applies the Phase 03 accepted-risk policy:

- Run `npm audit --omit=dev --json` in `Backend`.
- Run `npm audit --omit=dev --json` in `Frontend/Ecommerce-main/my-app`.
- Treat backend production vulnerabilities as blocking unless a future accepted-risk entry explicitly allows them.
- Keep frontend audit output visible.
- Allow only the existing CRA/react-scripts tooling-risk family already documented in `03-SECURITY-RISK-REGISTER.md`.
- Fail if the frontend audit contains new direct runtime/browser dependency findings outside that accepted family.
- Print a concise summary with backend status, frontend status, accepted frontend tooling-risk count, and blocking findings.

Prefer parsing JSON from `npm audit` over brittle prose matching. Do not run `npm audit fix`.

</action>
<verify>

```powershell
node scripts/ci/check-audits.mjs
```

</verify>
<acceptance_criteria>

- Backend production audit findings fail the script.
- Frontend accepted CRA/tooling findings are reported, not hidden.
- New frontend runtime dependency findings fail or require a risk-register update.
- The script exits with the correct code for CI.

</acceptance_criteria>
</task>
