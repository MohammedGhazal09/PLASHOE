# Phase 06 Plan 01: Admin Order Read API Foundation

<objective>
Add the protected admin order route foundation for listing and inspecting all orders, with strict query validation, shared pagination metadata, compact list payloads, limited user identity population, order query indexes, and authorization regression tests.
</objective>

## Recommendation

Build the admin order read path before fulfillment writes. This gives Phase 06 a protected `/api/admin/orders` surface, reusable pagination behavior, and regression tests proving customer order routes remain unchanged.

## Artifacts This Plan Produces

- `Backend/utils/adminListQuery.js` with pagination metadata, `escapeRegex`, and date filter helpers.
- `Backend/validators/adminOrder.js` with `adminOrderParamsSchema` and `adminOrderListQuerySchema`.
- `Backend/controllers/adminOrderController.js` with `listAdminOrders` and `getAdminOrder`.
- `Backend/routes/adminOrderRoutes.js` mounted in `Backend/app.js` under `/api/admin/orders`.
- `Backend/models/Order.js` indexes for status/payment/order-number/date admin filters.
- `Backend/test/admin-order.test.js` route tests for auth, list, filters, pagination metadata, detail payload, and customer ownership regression.

<threat_model>
Threats addressed: non-admin access to all-order data, accidental expansion of customer `/api/orders`, unbounded admin order queries, unsafe regex search, excessive user data exposure in admin order detail, and order list payloads that expose full item/user documents. This plan does not add fulfillment writes or admin UI pages.
</threat_model>

  <files>Backend/app.js; Backend/routes/adminOrderRoutes.js; Backend/controllers/adminOrderController.js; Backend/models/Order.js</files>
  <action>
  Add `Backend/routes/adminOrderRoutes.js` with `router.use(protect, admin)`, `GET /` using `validateRequest({ query: adminOrderListQuerySchema })`, and `GET /:id` using `validateRequest({ params: adminOrderParamsSchema })`. Mount the router in `Backend/app.js` as `app.use('/api/admin/orders', defaultJsonParser, adminOrderRoutes)` before the customer `/api/orders` mount. Implement `listAdminOrders` to filter by status, paymentStatus, created date range, and escaped `q`; `q` must match orderNumber or limited matching user ids found by user `name`/`email`. Return compact rows containing `_id`, `orderNumber`, limited `user` name/email, `status`, `paymentStatus`, `total`, `itemCount`, `trackingNumber`, `carrier`, `estimatedDeliveryDate`, `createdAt`, and `updatedAt`. Implement `getAdminOrder` to return the full order populated with user `name email` only. Add minimal `Order` schema indexes for `{ status: 1, paymentStatus: 1, createdAt: -1 }`, `{ orderNumber: 1 }`, and `{ user: 1, createdAt: -1 }`.
  </action>
  <verify>
  `cd Backend && npm test -- admin-order.test.js order.test.js`
  </verify>
  <acceptance_criteria>
  - `GET /api/admin/orders` is mounted and requires bearer auth plus admin role.
  - `GET /api/admin/orders/:id` is mounted and requires bearer auth plus admin role.
  - Customer `GET /api/orders` still returns only the current user's orders.
  - Admin list response uses the required pagination envelope.
  - Admin list rows do not include full user documents, passwords, or full current product documents.
  - Admin detail response includes user `name` and `email` and excludes password fields.
  </acceptance_criteria>
</task>
