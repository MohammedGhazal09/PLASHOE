# Phase 06 Plan 02: Fulfillment Transition and Tracking Workflow

<objective>
Add the protected admin fulfillment update path that advances eligible orders through `processing` to `shipped` to `delivered`, enforces Phase 05 payment state gates, captures tracking data, appends tracking history safely, and returns machine-readable conflicts for invalid domain states.
</objective>

## Recommendation

Keep fulfillment rules in one service and let the route/controller only validate input and return the service result. This prevents the admin HTTP layer from growing hidden payment and tracking business rules.

## Artifacts This Plan Produces

- `Backend/services/fulfillmentService.js` with `advanceOrderFulfillment`.
- `adminFulfillmentUpdateSchema` in `Backend/validators/adminOrder.js`.
- `updateAdminOrderFulfillment` in `Backend/controllers/adminOrderController.js`.
- `PATCH /api/admin/orders/:id/fulfillment` in `Backend/routes/adminOrderRoutes.js`.
- Extended `Backend/test/admin-order.test.js` coverage for valid transitions, conflicts, tracking history, no-op retries, and correction events.

<threat_model>
Threats addressed: shipping unpaid or refunded orders, skipping fulfillment states, client-replaced tracking history, duplicate tracking events from retries, stale same-status updates silently losing tracking corrections, and controllers embedding inconsistent payment-state rules. This plan does not add cancellation, refund initiation, or full admin UI operations.
</threat_model>

  <action>
  Create `Backend/services/fulfillmentService.js` exporting `advanceOrderFulfillment({ orderId, updates })`. Load the order by id and throw a `404`-style domain error with code `ORDER_NOT_FOUND` when missing. Permit shipping only when `paymentStatus` is `paid` or `not_required`; reject `requires_payment`, `payment_pending`, `payment_failed`, `payment_canceled`, `refunded`, and `partially_refunded` with status `409` and code `PAYMENT_NOT_SHIPPABLE`. Permit `processing` -> `shipped`, same-status `shipped` correction, and `shipped` -> `delivered`. Reject skipped, reversed, cancelled, pending, and delivered-to-earlier transitions with status `409` and code `INVALID_FULFILLMENT_TRANSITION`.
  </action>
  <verify>
  `cd Backend && npm test -- admin-order.test.js`
  </verify>
  <acceptance_criteria>
  - `processing` paid or not_required orders can be advanced to `shipped`.
  - `shipped` orders can be advanced to `delivered`.
  - `processing` -> `delivered` returns `409` with `INVALID_FULFILLMENT_TRANSITION`.
  - Unpaid, failed, canceled, refunded, and partially refunded orders cannot be shipped.
  - Missing order returns a stable not-found response and machine-readable `ORDER_NOT_FOUND` error.
  </acceptance_criteria>
</task>

<task type="code">
  <name>Apply tracking fields, timestamps, history events, no-op retries, and corrections</name>
  <read_first>
  Backend/services/fulfillmentService.js
  Backend/models/Order.js
  Backend/test/admin-order.test.js
  .planning/phases/06-admin-fulfillment-operations/06-CONTEXT.md
  </read_first>
  <files>Backend/services/fulfillmentService.js; Backend/test/admin-order.test.js</files>
  <action>
  In `advanceOrderFulfillment`, require `carrier` and `trackingNumber` before setting status `shipped`. Store `carrier`, `trackingNumber`, and optional `estimatedDeliveryDate` on the order. When the accepted update enters `shipped`, set `shippedAt` if absent and append one tracking history event with server `timestamp`, status `shipped`, optional `description`, and optional `location`. For same-status `shipped` updates, compare carrier, trackingNumber, estimatedDeliveryDate, description, and location against current stored values and the latest equivalent history input; return `200` no-op without appending when no tracked field changed. When a shipped correction changes tracked fields, append one new `shipped` history event. Before setting `delivered`, require current status `shipped` and complete carrier/trackingNumber; set `deliveredAt` if absent and append one `delivered` history event.
  </action>
  <verify>
  `cd Backend && npm test -- admin-order.test.js`
  </verify>
  <acceptance_criteria>
  - Setting `shipped` without `carrier` or `trackingNumber` returns `409` with `TRACKING_REQUIRED`.
  - Successful shipped transition sets `status`, `carrier`, `trackingNumber`, `shippedAt`, and one `trackingHistory` event.
  - Retrying the identical shipped update returns `200` and does not append a duplicate event.
  - Changing shipped tracking data appends one new shipped history event.
  - Delivered transition requires an existing shipped state with complete tracking fields, sets `deliveredAt`, and appends one delivered event.
  </acceptance_criteria>
</task>
