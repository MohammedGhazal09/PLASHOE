<objective>
Make every public product list endpoint use one bounded, validated backend catalog contract with supporting indexes and route tests.
</objective>

## Context

- Read `07-SPEC.md`, `07-CONTEXT.md`, `07-RESEARCH.md`, and `07-PATTERNS.md` before editing.
- Phase 03 already added strict product query validation and `limit <= 100` for `GET /api/products`.
- Phase 07 must close the gap where `/men`, `/women`, and `/sale` bypass pagination and response metadata.
- Product indexing belongs in the schema for this phase; production index-build operations are not in scope.

## Recommendation

Implement the backend contract first. The frontend plans can then rely on one canonical query/envelope instead of working around route-specific behavior.

## Artifacts This Plan Produces

- Shared list helper or equivalent controller path for all product list endpoints.
- Validated product list routes for root, men, women, and sale.
- Product schema indexes for catalog filters and sorts.
- `Backend/test/product.test.js`.
- Product endpoint docs update.

<threat_model>

### T-07-01: Unbounded Catalog Reads

Risk: legacy product list routes can return entire collections and create slow requests.

Mitigation: apply validated `page` and `limit` to every public product list route and add tests for default `20` and cap rejection.

### T-07-02: Query Contract Drift

Risk: route-specific response shapes force frontend branches and hide invalid query bugs.

Mitigation: return the same envelope for root, gender, and sale routes, including `page` and `limit`.

### T-07-03: Missing Query Support Indexes

Risk: common catalog filters and sorts become slow as product count grows.

Mitigation: add schema-level indexes that match approved Phase 07 query dimensions only.

</threat_model>
