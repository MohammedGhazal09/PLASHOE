<objective>
Finish Phase 07 by splitting mixed API wrappers, extracting only remaining touched catalog/API logic that needs a smaller boundary, updating docs, and running the full regression sweep.
</objective>

## Context

- Phase 06 explicitly deferred splitting `contactApi` and `couponApi` out of `ordersApi.js` to Phase 07.
- `Contact.jsx` currently imports `contactApi` from `ordersApi.js`.
- Existing wrapper tests demonstrate the expected Axios mock style.
- This plan should not reopen payment, checkout, admin fulfillment, or cart store behavior unless an import move requires a small test update.

## Recommendation

Complete the API module split after catalog normalization is in place, then use the final plan for docs and broad regression verification. This keeps source changes small and makes the final static checks meaningful.

## Artifacts This Plan Produces

- Resource-specific contact and coupon API modules.
- Order-only `ordersApi.js`.
- Updated imports/mocks/tests.
- Focused extraction cleanup for touched catalog/API code.
- Updated docs and final verification output.

<threat_model>

### T-07-07: Misleading API Ownership

Risk: contact/coupon calls remain hidden in an order module, causing import mistakes and weak wrapper coverage.

Mitigation: split wrappers by resource, test each wrapper, and assert no contact/coupon exports remain in `ordersApi.js`.

### T-07-08: Broad Cleanup Creep

Risk: "large page extraction" turns into unrelated checkout/account refactors.

Mitigation: touch checkout/account only for required import/test updates and prioritize catalog/product extraction completed in Plan 07-02.

### T-07-09: Docs Drift

Risk: API and development docs keep pointing developers at old wrapper paths and stale test commands.

Mitigation: update only the docs sections directly affected by product catalog contract and wrapper split.

</threat_model>
