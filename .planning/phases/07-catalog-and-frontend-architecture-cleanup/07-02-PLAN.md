<objective>
Move all storefront product loading through one normalized frontend catalog boundary and stop product UI from depending on legacy static JSON shapes.
</objective>

## Context

- This plan depends on the backend list envelope from Plan 07-01.
- Current pages duplicate API-first/static-fallback loading and parse legacy product shapes differently.
- `ProductCard` and `QuickViewModal` are the cart-entry points and must remain compatible with the Phase 04 cart-store boundary.
- This is not a visual redesign; keep existing layout and copy unless required by data/control changes.

## Recommendation

Put normalization and loading behind a service/hook first, then update pages and shared product components to consume that normalized contract. This keeps server state local to catalog pages and avoids a global catalog store.

## Artifacts This Plan Produces

- `normalizeProduct.js` and focused tests.
- `catalogService.js` and `useCatalogProducts.js`.
- Updated catalog pages and Home product sections.
- Updated `ProductGrid`, `ProductCard`, and `QuickViewModal`.
- Frontend tests that prove fallback, query, pagination, and product display behavior.

<threat_model>

### T-07-04: Product Shape Drift

Risk: backend and static fallback records keep diverging, forcing defensive UI code and hidden cart bugs.

Mitigation: normalize both sources once and test backend, fallback, missing-price, sale, and image cases.

### T-07-05: Hidden Backend Failures

Risk: frontend falls back to static products when the backend returns a valid empty filtered result.

Mitigation: fallback only on request failure; valid empty responses render empty state.

### T-07-06: Client-Side Full Dataset Filtering

Risk: catalog pages load too many products and filter locally even when backend supports the query.

Mitigation: backend-backed query changes call `productsApi.getAll(params)`; client filtering stays fallback-only.

</threat_model>
