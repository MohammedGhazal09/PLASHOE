<objective>
Run the required local pre-deploy gates, then record real hosted backend and frontend staging smoke evidence or honest blockers.
</objective>

## Context

- This plan depends on `09-01` because smoke evidence needs the setup and verification tables.
- Hosted smoke checks require user-provided staging URLs and external host setup.
- Local command gates must pass before hosted evidence can close Phase 09.
- CRA staging frontend config must be proved at runtime because build-time `REACT_APP_API_URL` can drift from host dashboard expectations.

## Recommendation

Treat this plan as the staging proof gate. It should either produce concrete hosted evidence or explicitly block with missing external setup, not pass on local-only checks.

## Artifacts This Plan Produces

- Updated `09-USER-SETUP.md` with staging backend/frontend origins, deployment status, and any blockers.
- Updated `09-VERIFICATION.md` with local gate outcomes and hosted backend/frontend smoke evidence.

<threat_model>

### T-09-04: Local Gates Are Skipped Before Deployment

Risk: hosted smoke checks run against unverified code and hide regressions.

Mitigation: block hosted acceptance until backend tests, frontend tests/build, static checker, and audit policy are recorded.

### T-09-05: Health Is Mistaken For Readiness

Risk: `/api/health` returns 200 while MongoDB-backed routes are unusable.

Mitigation: require `/api/ready` status 200 with `ready: true` and MongoDB `connected`.

### T-09-06: Frontend Calls Localhost In Staging

Risk: the deployed static frontend was built with a local API URL and appears loaded but cannot use staging APIs.

Mitigation: require browser network evidence or equivalent runtime proof that API calls use the staging backend base URL.

</threat_model>
