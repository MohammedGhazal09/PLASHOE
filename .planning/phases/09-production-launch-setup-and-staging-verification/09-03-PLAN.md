<objective>
Prove Stripe test-mode checkout/webhook delivery against staging, run the final secret-free evidence scan, and close Phase 09 only if every required launch acceptance criterion has passed.
</objective>

## Context

- This plan depends on `09-01` setup evidence and `09-02` hosted staging smoke evidence.
- PLASHOE already has local webhook tests and raw-body route wiring; Phase 09 must prove hosted Stripe dashboard delivery to the staging backend.
- Stripe test mode is required. Live customer charges and production cutover are Phase 12.

## Recommendation

Use dashboard-visible Stripe delivery evidence plus sanitized backend no-sustained-5xx evidence. Keep refund events subscribed, but do not block the phase on refund smoke unless a safe test refund path exists.

## Artifacts This Plan Produces

- Updated `09-USER-SETUP.md` with Stripe endpoint, test-mode status, selected events, and secret-manager status.
- Updated `09-VERIFICATION.md` with Stripe delivery evidence, final secret scan, final blocker table, and final Phase 09 verdict.

<threat_model>

### T-09-07: Stripe Webhook Proof Is Only Local

Risk: tests pass locally but the hosted staging endpoint is unreachable or misconfigured in Stripe.

Mitigation: require test-mode dashboard delivery evidence to the hosted staging endpoint.

### T-09-08: Webhook Evidence Leaks Payloads Or Secrets

Risk: copying dashboard payloads or signing secrets into planning docs exposes payment data or credentials.

Mitigation: record endpoint URL, event names, delivery status, status codes, and sanitized summaries only.

### T-09-09: Phase Closes With Remaining Blockers

Risk: setup rows, smoke checks, or Stripe events remain pending while the phase is marked passed.

Mitigation: final acceptance coverage table must have no failed or blocked required rows before frontmatter status becomes passed.

</threat_model>
