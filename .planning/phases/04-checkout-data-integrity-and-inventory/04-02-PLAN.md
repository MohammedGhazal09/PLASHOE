# Phase 04 Plan 02: Inventory, Coupon, and Cancellation Consistency

<objective>
Layer inventory, coupon max-use, and cancellation stock-restore rules onto the transactional checkout foundation from `04-01` so product stock and coupon usage cannot drift under conflicts, failures, retries, or cancellation.
</objective>

## Recommendation

Keep cart stock checks as pre-mutation validation only, then enforce stock and coupon limits again at checkout with conditional writes inside the same transaction. Use structured `409` conflicts for all state-race failures so frontend handling can stay consistent in Wave 3.

## Artifacts This Plan Produces

- Cart add/update `409` stock validation and unchanged-cart tests.
- Checkout product aggregation and conditional stock decrements.
- Checkout coupon max-use recheck and conditional `usedCount` increment.
- Cancellation transaction that restores stock exactly once.
- Backend tests for stock decrement, stock conflict, missing product conflict, coupon limit/concurrency, forced rollback including stock/coupon, and cancellation restore.

<threat_model>
Threats addressed: overselling, stale cart checkout, deleted product checkout, exhausted coupon use under concurrency, double coupon consumption on retry, stock not restored on cancellable orders, and double stock restoration on repeated cancellation.
</threat_model>

<task type="code">
  <name>Enforce stock during checkout</name>
  <read_first>
  Backend/services/checkoutService.js
  Backend/models/Product.js
  Backend/models/Cart.js
  Backend/test/order.test.js
  </read_first>
  <files>Backend/services/checkoutService.js; Backend/test/order.test.js; Backend/test/helpers/factories.js</files>
  <action>
  In checkout, aggregate cart quantities by product, verify populated products still exist, and conditionally decrement each product with `{ _id: productId, stock: { $gte: requested } }` and `$inc: { stock: -requested }` inside the same transaction as order creation and cart clearing. If any product is missing or insufficient, abort the whole checkout with `409`, no order, intact cart, unchanged coupon usage, and unchanged stock.
  </action>
  <verify>
  `cd Backend && npm test -- order.test.js`
  </verify>
  <acceptance_criteria>
  - Successful checkout decrements stock by ordered quantities exactly once.
  - Insufficient checkout stock returns `409`, creates no order, preserves cart, and does not consume coupons.
  - Missing/deleted product checkout returns `409` and preserves state.
  - Exact idempotent retry does not decrement stock again.
  </acceptance_criteria>
</task>

  <files>Backend/services/checkoutService.js; Backend/test/order.test.js; Backend/test/helpers/factories.js</files>
  <action>
  Re-read the applied coupon in the checkout transaction. Enforce `isActive`, `validFrom`, `validUntil`, `minOrderAmount`, and `maxUses` before order persistence. Increment `usedCount` with a condition that prevents exceeding `maxUses`. If the condition fails, abort checkout with `409` and do not create an order, clear a cart, decrement stock, or consume coupon usage.
  </action>
  <verify>
  `cd Backend && npm test -- order.test.js`
  </verify>
  <acceptance_criteria>
  - Coupon at max use returns `409` during checkout.
  - Concurrent attempts against a one-use coupon cannot persist more successful orders than the coupon allows.
  - Failed checkout and exact idempotent retry do not double-increment `usedCount`.
  - Cart coupon application can remain a preliminary validation, but checkout is authoritative.
  </acceptance_criteria>
</task>
