# Phase 04 Plan 01: Transactional Checkout and Idempotency Foundation

<objective>
Create the backend foundation that makes checkout one transactionally consistent operation and makes browser/network retries safe before inventory-specific rules are layered in Wave 2.
</objective>

## Recommendation

Build a small checkout service with explicit session passing and deterministic tests. Require `Idempotency-Key` on checkout, but handle replays before empty-cart rejection so a successful checkout can be safely retried after the cart has been cleared.

## Artifacts This Plan Produces

- `Backend/test/setup.js` using `MongoMemoryReplSet` and `wiredTiger`.
- `Backend/services/checkoutService.js` for transaction, idempotency, fingerprint, and failpoint-test seams.
- Updated `Backend/models/Order.js` with `idempotencyKey`, `cartFingerprint`, a scoped unique index, and crypto-backed `PLS-` order numbers.
- Updated `Backend/controllers/orderController.js` delegating checkout to the service.
- Extended `Backend/test/order.test.js` coverage for missing idempotency header, exact retry, stale-key conflict, forced rollback, and order-number uniqueness.

<threat_model>
Threats addressed: duplicate checkout submission, partial checkout persistence after mid-flow errors, stale idempotency-key reuse, order-number collision under concurrent creates, and transaction tests that falsely pass on a standalone MongoDB topology. This plan does not yet enforce product stock or coupon max-use concurrency; Wave 2 owns those rules.
</threat_model>

  Create `Backend/services/checkoutService.js` with `createCheckoutFromCart({ userId, shippingAddress, notes, idempotencyKey, hooks })`. The service must validate the idempotency key, compute a deterministic fingerprint from sorted cart items, coupon state, shipping address, and notes, check existing orders for exact replay or stale-key conflict, then run order creation, coupon increment, and cart clearing in one Mongoose transaction with explicit session passing. Keep operations sequential inside the transaction. Let tests inject optional hooks after each write point to force rollback without exposing public failpoints.
  </action>
  <verify>
  `cd Backend && npm test -- order.test.js`
  </verify>
  <acceptance_criteria>
  - `orderController.createOrder` delegates checkout orchestration to the service.
  - Missing `Idempotency-Key` returns a stable `400` contract error.
  - First successful checkout returns `201`.
  - Exact retry with the same key returns `200` and the existing order without clearing or incrementing again.
  - Reusing the same key for changed non-empty cart/request state returns `409` with an `errors` array.
  - Forced failures after order creation, coupon increment, or cart clearing roll back visible state.
  </acceptance_criteria>
