# Phase 04: checkout-data-integrity-and-inventory - Context

**Gathered:** 2026-06-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 04 makes authenticated checkout consistent, stock-aware, and retry-safe across orders, carts, coupons, inventory, and order display numbers. It also normalizes frontend cart item data at the Zustand store boundary so checkout and cart UI components stop depending on mixed backend/local item shapes.

This phase hardens the current non-payment checkout path only. It does not add payment provider behavior, guest order creation, admin fulfillment, per-size inventory, reservation expiry, or broad catalog/product normalization.

</domain>

<decisions>
## Implementation Decisions

### Skill and Execution Boundaries
- **D-01:** Use installed local skills as supporting guidance: `mongodb`, `express-rest-api`, `zustand-state-management`, `testing-quality-standards`, and current GSD workflow rules are enough for this phase.
- **D-02:** Do not install duplicate external skills found during `find-skills` search unless later planning uncovers a concrete missing capability. The external MongoDB candidates were lower-value duplicates for this stack.
- **D-03:** Do not rely on subagent fan-out while the repo instruction says not to use subagents. Execute planning/research inline unless the user later changes that instruction.

### Checkout Atomicity and Retry Safety
- **D-04:** Implement checkout with a Mongoose session transaction plus idempotency protection. The transaction prevents partial order/cart/coupon/stock writes; idempotency prevents duplicate browser/network retries.
- **D-05:** Extract a focused backend checkout service/helper called by `Backend/controllers/orderController.js` instead of growing the controller with all cross-model orchestration.
- **D-06:** Use an `Idempotency-Key` HTTP header generated by the frontend and scoped to the authenticated user. Do not add retry identity to the shipping-address request body.
- **D-07:** Store retry state on the order using `idempotencyKey` plus a `cartFingerprint`, with a compound uniqueness guard scoped to the user.
- **D-08:** First successful checkout returns `201`. An exact retry with the same idempotency key and same cart fingerprint returns `200` with the existing order. Reusing the same key against changed cart state returns `409`.
- **D-09:** The cart fingerprint should bind the idempotency key to the exact cart contents, coupon, and relevant cart state so stale key reuse cannot create or return the wrong order.

### MongoDB Topology and Transaction Tests
- **D-10:** Update backend transaction tests to use `MongoMemoryReplSet` instead of the current single-node `MongoMemoryServer` harness.
- **D-11:** Treat MongoDB replica-set or Atlas-compatible transaction support as required for checkout integrity. If a transaction cannot start, checkout must fail before mutating order, cart, coupon, or stock state.
- **D-12:** Keep the existing Vitest/Supertest route-test style. Add service-level seams or spies to force failures after individual checkout write points; do not expose public failpoint routes or query parameters.

### Inventory and Cart Stock Behavior
- **D-13:** Cart add/update validates stock but does not reserve or decrement inventory. Actual stock decrement happens only during checkout.
- **D-14:** Use the existing aggregate `Product.stock` field across all sizes for Phase 04. Per-size inventory is deferred.
- **D-15:** Cart stock conflicts return `409` with item-level details and leave the cart unchanged. Do not auto-reduce quantities or silently mutate customer cart state.
- **D-16:** Checkout aggregates requested quantities by product and uses conditional atomic stock decrement operations inside the transaction.
- **D-17:** If any cart item is missing, deleted, or no longer has enough stock at checkout time, reject the whole checkout with `409`, create no order, and keep the cart intact.

### Coupon, Pricing, and Cancellation Consistency
- **D-18:** Preserve server-stamped `priceAtAdd` for Phase 04 checkout pricing. Do not introduce current-price repricing or a promotion/pricing engine in this phase.
- **D-19:** Recheck coupon active/date/max-use rules during checkout and conditionally increment `usedCount` inside the same transaction as order creation, cart clearing, and stock decrement.
- **D-20:** Failed checkout attempts and duplicate idempotent retries must not consume coupon usage.
- **D-21:** Cancellation restores stock exactly once when an owned order transitions from `pending` or `processing` to `cancelled`. Repeat cancellation of an already-cancelled order must not restore stock again. Existing shipped/delivered rejection behavior stays.

### Order Numbers and Conflict Contracts
- **D-22:** Replace the current `Date.now()` plus `countDocuments()` order-number hook with a collision-safe `PLS-` time/random suffix generated with Node crypto and protected by the existing unique index.
- **D-23:** Do not require strict sequential display numbers in Phase 04. Uniqueness and `PLS-` prefix preservation are the required guarantees.
- **D-24:** New domain conflicts use `409` with `{ success: false, message, errors: [...] }`.
- **D-25:** Conflict error entries should be machine-readable and may include `code`, `resource`, `productId`, `cartItemId`, `couponCode`, `checkoutAttempt`, `requested`, and `available`.

### Frontend Cart Normalization
- **D-26:** Normalize mixed backend/local cart item shapes at the Zustand cart-store boundary, not separately inside every component.
- **D-27:** The normalized cart view model should expose stable fields such as `id`, `cartItemId`, `productId`, `name`, `image`, `size`, `quantity`, `unitPrice`, `lineTotal`, and source/raw metadata when needed for mutations.
- **D-28:** `Checkout.jsx`, `Cart.jsx`, and `CartSidebar.jsx` should read normalized items and remove page/component-level mixed-shape fallback helpers.
- **D-29:** Tolerate existing persisted `cart-storage` shapes on hydration/actions, then write normalized items going forward. Do not wipe guest carts solely for this migration.
- **D-30:** Keep frontend scope limited to `cartStore`, checkout/cart/cart-sidebar consumers, and focused tests. Broader product/catalog normalization remains Phase 07.
- **D-31:** On checkout/cart conflict responses, the frontend should preserve the cart, show the conflict message, and sync cart state after the failure.

### Verification and Documentation
- **D-32:** Backend coverage must include transaction success, forced failure rollback, idempotent retry, stale idempotency-key conflict, stock conflict, stock decrement, cancellation restore, coupon limit concurrency, and concurrent order-number uniqueness.
- **D-33:** Frontend coverage must include normalized items for backend sync and local guest mutations, plus checkout/cart UI behavior using normalized item fields.
- **D-34:** Phase completion must update `docs/API.md` and `docs/TESTING.md`, then verify with backend tests, frontend tests, frontend build, and the retained static contract checker.

### Agent Discretion
- The planner/executor may choose exact service/helper filenames if the controller remains thin and tests can target failure points deterministically.
- The exact cart fingerprint algorithm may vary if it is stable for equivalent cart state and changes when items, quantities, sizes, coupon, or relevant cart metadata change.
- The exact crypto suffix format for `PLS-` order numbers may vary if it is concurrency-safe, unique-index-backed, and dependency-free.
- The exact normalized item metadata fields may vary if all cart consumers get stable display and mutation fields without mixed-shape fallback logic.

</decisions>

## Existing Code Insights

### Reusable Assets
- `Backend/app.js`: already supports Supertest route tests without opening the production server.
- `Backend/test/setup.js`: central database harness; it currently uses `MongoMemoryServer` and should become replica-set capable for transaction tests.
- `Backend/test/helpers/factories.js`: existing user/product/cart/coupon/order factories can be extended for stock, coupon limit, idempotency, and cancellation scenarios.
- `Backend/validators/cart.js` and `Backend/validators/order.js`: existing strict Zod boundaries should be preserved while adding stock/idempotency behavior around validated data.
- `Frontend/Ecommerce-main/my-app/src/store/cartStore.js`: single best boundary for normalizing backend and guest cart items.
- `Frontend/Ecommerce-main/my-app/src/api/axios.js`: shared HTTP layer already attaches auth and can support idempotency header propagation through the order API wrapper.

### Established Patterns
- Backend code uses ES modules, thin route files, controllers for request behavior, and Mongoose models for persistence shape.
- Backend error envelopes center on `{ success, message, data }`, with Phase 03 allowing optional `errors` arrays for structured validation/conflict details.
- Cart and order routes are authenticated with `router.use(protect)`, so Phase 04 remains authenticated-only.
- Frontend state uses Zustand with persisted cart/auth stores; cart normalization should happen in the store rather than in page components.
- Frontend tests stay on CRA/Jest/React Testing Library; backend tests stay on Vitest/Supertest.

### Integration Points
- Checkout service connects `Order`, `Cart`, `Product`, and `Coupon` inside one Mongoose session transaction.
- Idempotency connects frontend `ordersApi.create`, backend order route handling, and new order persistence fields/indexes.
- Stock validation connects cart add/update, checkout order creation, and cancellation restore.
- Conflict contracts connect backend `409` response bodies to checkout/cart UI handling and documentation.
- Cart normalization connects backend cart sync, local guest cart actions, selectors, checkout page, cart page, sidebar, and persisted `cart-storage` compatibility.
