# Phase 03 Plan 01: Security Middleware and Startup Configuration Baseline

<objective>
Add the backend security baseline that must exist before validators and dependency remediation: security headers, rate limits, explicit JSON body caps, fail-fast runtime config validation, shorter JWT defaults, explicit JWT algorithms, and stable security error envelopes.
</objective>

## Recommendation

Build this as small Express modules, not a security framework migration. Keep app import safe for tests, validate env only in startup/config tests, and keep all response envelopes compatible with the existing frontend `{ success, message }` convention.

## Artifacts This Phase Produces

- New `Backend/config/env.js` with `validateRuntimeEnv` and safe config parsing.
- New `Backend/config/security.js` with rate-limit, body-limit, and JWT constants.
- New `Backend/middleware/security.js` with Helmet setup, limiters, JSON parser choices, and security error formatting.
- Updated `Backend/app.js` middleware order.
- Updated `Backend/server.js` startup validation before `connectDB()` and `app.listen(...)`.
- Updated `Backend/middleware/auth.js` JWT verification options.
- Updated auth token signing defaults in `Backend/controllers/authController.js`.
- Backend tests for config validation, rate limits, body caps, JWT defaults, and stable security envelopes.

<threat_model>
Threats addressed: unauthenticated request bursts, oversized JSON denial-of-service attempts, missing or weak runtime secrets, long-lived browser bearer tokens, JWT algorithm confusion, and raw error disclosure. This plan does not introduce refresh tokens, cookies, Redis-backed rate limiting, or a full auth rewrite.
</threat_model>
