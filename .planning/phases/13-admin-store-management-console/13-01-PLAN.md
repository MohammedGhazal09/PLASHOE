<objective>
Create the protected admin entry point: a frontend admin-only route guard, `/admin` route shell, section navigation, and focused authorization tests.
</objective>

## Context

- `13-SPEC.md` locks the requirement that admin routes are accessible only to authenticated admins.
- `13-CONTEXT.md` requires a dedicated admin guard and says backend authorization remains the real security boundary.
- The app currently has protected customer routes but no admin route.
- `authStore.js` already persists the backend user object, including `isAdmin`.

## Recommendation

Start with the access boundary and shell before building any resource screens. Every later admin screen depends on a working admin route structure.

## Artifacts This Plan Produces

- `Frontend/Ecommerce-main/my-app/src/components/AdminRoute.jsx`
- `Frontend/Ecommerce-main/my-app/src/components/AdminRoute.test.jsx`
- `Frontend/Ecommerce-main/my-app/src/pages/AdminConsole.jsx`
- Updated `App.js`, `pages/index.js`, and optionally `Header.jsx`.

<threat_model>

### T-13-01: Non-admin Console Exposure

Risk: an authenticated customer can render admin screens if the route only checks login.

Mitigation: create a dedicated admin guard that checks `user.isAdmin === true` and cover all three states in tests.

### T-13-02: Navigation Mistaken for Security

Risk: hiding an admin link in the header is treated as authorization.

Mitigation: protect the route regardless of navigation visibility and retain backend middleware enforcement.

</threat_model>
