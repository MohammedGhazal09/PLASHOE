<objective>
Replace CRA/react-scripts with Vite while preserving the frontend's public config names, static build output location, Tailwind styling pipeline, and public asset behavior.
</objective>

## Context

- Phase 10 SPEC and CONTEXT lock Vite/Vitest migration and explicitly exclude React, React Router, Tailwind, TypeScript, and feature changes.
- Current `npm run build` passes under CRA but emits the known `OrderDetail.jsx`, Node deprecation, stale Browserslist, and CRA deployment warning surface.
- Vite docs support root `vite.config.js`, `envPrefix`, `publicDir`, and `build.outDir`.
- Current source reads `REACT_APP_*` through `src/config/config.js`, but runtime public asset paths still use CRA `PUBLIC_URL`.

## Recommendation

Do the toolchain and runtime compatibility work first, then let the test/warning plan convert the test suite against the new runner. Keep this plan focused on package/config/app-shell compatibility so failures localize to Vite migration rather than test rewrites.

<threat_model>

### T-10-01: Tooling Debt Remains in Production Graph

Risk: keeping `react-scripts` or test-only packages in `dependencies` leaves production audit findings reachable and defeats `TOOL-01`.

Mitigation: remove `react-scripts` and `web-vitals`; install Vite/Vitest/test packages as dev dependencies; verify package and lockfile references.

### T-10-02: Public Config Leak or Rename

Risk: migrating env handling carelessly can either expose unprefixed secrets or break existing `REACT_APP_*` deployment variables.

Mitigation: set non-empty `envPrefix: "REACT_APP_"` and route code through a small env helper.

### T-10-03: Asset Path Drift

Risk: replacing CRA `PUBLIC_URL` with ad hoc strings can break fallback catalog JSON and cart/product images on subpath deploys.

Mitigation: use one `import.meta.env.BASE_URL` helper and update every `PUBLIC_URL` call site through it.

</threat_model>
