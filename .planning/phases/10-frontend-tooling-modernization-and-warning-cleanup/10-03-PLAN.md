<objective>
Remove the CRA/react-scripts accepted audit policy, update CI/docs to the new Vite/Vitest command contract, and prove the same release gate shape still passes.
</objective>

## Context

- Plan 10-01 removes CRA/react-scripts from the frontend package and lockfile.
- Plan 10-02 makes the frontend Vitest suite and Vite build warning-clean.
- Current `scripts/ci/check-audits.mjs` accepts CRA/react-scripts reachability and prints `Accepted frontend CRA/tooling findings`.
- Current CI frontend job runs `npm test -- --watchAll=false`.
- Active docs still describe CRA/react-scripts, `PUBLIC_URL`, `--watchAll=false`, and accepted CRA audit debt.

## Recommendation

Make audit policy strict after the lockfile no longer contains CRA. Then update CI/docs and run the full gate once so the migration is proven as a release-readiness improvement, not only a local package change.

## Artifacts This Plan Produces

- Simplified `scripts/ci/check-audits.mjs` frontend audit policy.
- Updated `scripts/ci/check-audits.test.mjs` policy tests.
- Updated `.github/workflows/ci.yml` frontend test command.
- Updated active docs for Vite/Vitest commands, env behavior, audit policy, and deployment build behavior.

<threat_model>

### T-10-07: Stale Risk Acceptance Survives

Risk: keeping CRA acceptance code or messaging makes new frontend vulnerabilities appear intentionally accepted.

Mitigation: remove CRA reachability helpers and accepted-count output; test that frontend findings block.

### T-10-08: CI Still Runs Old Runner Flags

Risk: CI can hang or fail if it passes CRA-only `--watchAll=false` to a new runner contract.

Mitigation: set frontend CI to `npm test` and keep `npm run build` as the build gate.

### T-10-09: Docs Reintroduce Old Commands

Risk: developers follow stale CRA docs and generate noisy or incorrect verification evidence.

Mitigation: update active docs and run targeted `rg` checks for old active instructions.

</threat_model>
