# Phase 05 Plan 03: Webhook Reconciliation and Refund Handling

<objective>
Implement the authoritative Stripe webhook path that verifies raw-body signatures, processes selected events idempotently, transitions order payment state, restores inventory once for terminal unpaid states, and records full/partial refunds.
</objective>

## Recommendation

Treat webhooks as the single source of truth for payment completion. Verify first, dedupe second, reconcile third, then mark the event processed only after the local state change succeeds.

## Artifacts This Plan Produces

- `Backend/routes/webhookRoutes.js` mounted at `/api/webhooks/stripe`.
- `Backend/controllers/webhookController.js` with signed event verification, event dispatch, order resolution, retryable failure handling, and duplicate event no-op behavior.
- Extended `paymentState.js` use for success, failure, cancellation/expiry, full refund, and partial refund transitions.
- `Backend/test/payment-webhook.test.js` with signed fixtures through the real Express route.

  <files>Backend/controllers/webhookController.js; Backend/services/paymentState.js; Backend/models/PaymentEvent.js; Backend/test/payment-webhook.test.js</files>
  <action>
  Implement event dispatch for `checkout.session.completed`, `checkout.session.expired`, `payment_intent.succeeded`, and `payment_intent.payment_failed`. Resolve the local order from event metadata first; when metadata is missing, retrieve the related Checkout Session or PaymentIntent through `paymentProvider`. Use `PaymentEvent` to skip duplicates. On success, set payment paid fields and fulfillment `processing`. On failure, set `payment_failed` with provider failure reason when available and restore inventory once. On expired Checkout Session, set `payment_canceled` and restore inventory once. If no single local order can be safely resolved, return a retryable `500` and do not mark the event processed.
  </action>
  <verify>
  `cd Backend && npm test -- payment-webhook.test.js order.test.js`
  </verify>
  <acceptance_criteria>
  - `payment_intent.succeeded` or `checkout.session.completed` marks the order `paid`, sets `paidAt`, persists provider intent id, and moves fulfillment `status` to `processing`.
  - `payment_intent.payment_failed` marks the order `payment_failed`, stores a failure reason when present, and restores stock once.
  - `checkout.session.expired` marks the order `payment_canceled` and restores stock once.
  - Duplicate event ids return success without changing payment fields or stock a second time.
  - Unresolved events return `500` and do not persist a processed event.
  </acceptance_criteria>
</task>
