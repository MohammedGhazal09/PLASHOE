# Phase 05 Plan 04: Frontend Payment Redirect and Return States

<objective>
Replace demo checkout UI with hosted payment redirect behavior, add protected return-state pages that refetch authoritative order state, and surface payment status on account/order detail screens.
</objective>

## Recommendation

Keep UI changes utilitarian and small. Do not add a Stripe frontend SDK or publishable key; hosted Checkout only needs a backend-returned URL and browser navigation.

## Artifacts This Plan Produces

- Updated `Checkout.jsx` with demo copy removed and `payment.checkoutUrl` redirect behavior.
- New `CheckoutReturn.jsx` exported from `pages/index.js` and mounted under protected `/checkout/success` and `/checkout/cancel`.
- Payment status labels on account order cards and order detail pages.
- Frontend tests for checkout redirect, failure preservation, return-state success/cancel/failure, and cancellation guard behavior.

  <action>
  Add one reusable `CheckoutReturn` page that accepts a `variant` prop of `success` or `cancel`, reads `orderId` and `session_id` from the URL query string, refetches the order through `ordersApi.getById(orderId)`, and renders customer-facing state from the fetched `paymentStatus`. Mount `/checkout/success` and `/checkout/cancel` under `ProtectedRoute`. If `orderId` is absent or refetch fails, show a concise recovery state with links/buttons to account orders and collection. The page must not treat the Stripe redirect query as authoritative proof of payment.
  </action>
  <verify>
  `cd Frontend/Ecommerce-main/my-app && npm test -- CheckoutReturn.test.jsx --watchAll=false`
  </verify>
  <acceptance_criteria>
  - `/checkout/success` and `/checkout/cancel` are protected routes.
  - Return page calls `ordersApi.getById(orderId)` before rendering final payment state.
  - Paid orders show `Paid`; pending orders show `Payment pending`; failed/canceled orders show the corresponding label.
  - Missing `orderId` and failed refetch paths show recovery actions instead of crashing.
  - Tests cover success, cancel, pending/failure labels, and missing order id.
  </acceptance_criteria>
</task>
