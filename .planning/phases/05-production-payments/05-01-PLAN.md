# Phase 05 Plan 01: Payment State, Configuration, and Provider Seam

<objective>
Create the backend payment foundation: Stripe dependency/config validation, independent order payment fields, durable event-id storage, and a small provider/state seam that later checkout and webhook plans can use without putting Stripe logic in controllers.
</objective>

## Context

Read these before editing:

- @Backend/test/helpers/factories.js

## Recommendation

Start with the state/config seam because every later payment behavior depends on the enum, provider identifiers, and runtime validation. Use the official Stripe SDK in the backend only; keep test values obviously fake.

## Artifacts This Plan Produces

- `Backend/package.json` and `Backend/package-lock.json` with the official Stripe Node SDK.
- `Backend/config/env.js` parsing `PAYMENTS_ENABLED`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `PAYMENT_SUCCESS_URL`, and `PAYMENT_CANCEL_URL`.
- `Backend/models/Order.js` with payment fields: `paymentStatus`, `paymentProvider`, `paymentProviderSessionId`, `paymentProviderIntentId`, `paidAt`, `paymentFailureReason`, `refundedAt`, and `refundAmount`.
- `Backend/models/PaymentEvent.js` with unique provider/event id storage.
- `Backend/services/paymentProvider.js` with `createCheckoutSession`, `constructWebhookEvent`, `retrieveCheckoutSession`, and `retrievePaymentIntent` exports.
- `Backend/services/paymentState.js` with transition helpers for paid, failed, canceled, refunded, and partially refunded orders.
